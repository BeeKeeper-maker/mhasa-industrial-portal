    48→### C1. `ignoreBuildErrors` and disabled React Strict Mode hide 23 TypeScript errors
    49→- **File**: `next.config.ts:5-8`
    50→- **Problem**: 
    51→  ```ts
    52→  typescript: { ignoreBuildErrors: true },
    53→  reactStrictMode: false,
    54→  ```
    55→  `bunx tsc --noEmit` reports **23 errors in `src/`** (plus 4 in `examples/` and `skills/` which are correctly excluded from the build but not from `tsconfig.json`'s `include` glob). Because the build ignores these, the production bundle ships with type holes that produce runtime bugs.
    56→- **Impact**: Production runtime defects that TypeScript was supposed to catch. Examples already in the codebase:
    57→  - `src/components/views/home-view.tsx:708` uses `pick()` but it is **never destructured from `useLocale()`** in `CTASection` → runtime `ReferenceError` when locale is Arabic.
    58→  - `src/components/views/home-view.tsx:145` references `current.ctaTextAr` which is **not on `HeroSlideDTO`** → undefined at runtime; Arabic CTA falls back silently.
    59→  - `src/components/admin/charts/index.tsx:275,323` pass `fetchStats` directly as a queryFn (signature mismatch) — Recharts incorrectly typed as `{}`; `data.contentCounts` and `data.topCategories` will throw at runtime when admin opens the Overview tab.
    60→  - `src/components/admin/admin-dashboard.tsx:135-136` passes `"settings"` and `"activity"` to `ResourceManager` but `ResourceKey` excludes those values — runtime comparison in `resource-manager.tsx:102,105` always falls through.
    61→- **Fix**: 
    62→  1. Set `typescript.ignoreBuildErrors: false` and `reactStrictMode: true` in `next.config.ts`.
    63→  2. Add `eslint: { ignoreDuringBuilds: false }` (default — remove any override).
    64→  3. Fix all 23 TypeScript errors (see "Files to Create/Modify Summary" below).
    65→  4. Re-run `bunx tsc --noEmit` until clean.

    67→### C2. Hardcoded NextAuth secret fallback + missing env var
    68→- **File**: `src/lib/auth.ts:66`, `.env` (only contains `DATABASE_URL`)
    69→- **Problem**: 
    70→  ```ts
    71→  secret: process.env.NEXTAUTH_SECRET ?? "mhasa-dev-secret-change-in-production-2024",
    72→  ```
    73→  `.env` does **not** define `NEXTAUTH_SECRET`, so production silently uses the public dev secret. Anyone who reads the public GitHub repo can forge a valid JWT and access the admin API.
    74→- **Impact**: Complete authentication bypass. **CVSS ~9.8 (Critical).**
    75→- **Fix**:
    76→  1. Remove the `?? "..."` fallback so the app **fails fast** if `NEXTAUTH_SECRET` is unset:
    77→     ```ts
    78→     const secret = process.env.NEXTAUTH_SECRET;
    79→     if (!secret) throw new Error("NEXTAUTH_SECRET must be set");
    80→     ```
    81→  2. Generate a 64-byte random hex secret (`openssl rand -hex 64`) and add it to `.env` / deployment env vars.
    82→  3. Add `.env.example` documenting all required vars (`DATABASE_URL`, `NEXTAUTH_SECRET`, `NEXTAUTH_URL`, `NODE_ENV`).
    83→  4. Run `git log --all -p -- .env` to confirm the secret was never committed; if it was, rotate immediately.

   105→### C4. File upload endpoint accepts any MIME / any size, stored as base64 in SQLite
   106→- **File**: `src/app/api/careers/apply/route.ts:20-33`
   107→- **Problem**:
   108→  ```ts
   109→  const resumeFile = formData.get("resume") as File | null;
   110→  if (resumeFile && resumeFile.size > 0) {
   111→    const buffer = Buffer.from(await resumeFile.arrayBuffer());
   112→    resumeUrl = `data:${resumeFile.type};base64,${buffer.toString("base64")}`;
   113→  }
   114→  ```
   115→  No MIME-type whitelist, no size limit, no virus scan, no extension validation. The file is then persisted into a SQLite TEXT column via Prisma (`JobApplication.resumeUrl`). A malicious 100 MB upload is fully buffered in memory and then base64-encoded (33% larger) before being written to the DB.
   116→- **Impact**:
   117→  - **Denial of service**: a single 50 MB upload will OOM the Node process and bloat the SQLite file.
   118→  - **Stored XSS via SVG**: an attacker can upload `evil.svg` containing script; if the admin downloads/views it inline in a browser, script executes in the admin origin.
   119→  - **DB bloat**: base64 resumes grow the SQLite file indefinitely with no cleanup path.
   120→- **Fix**:
   121→  1. Whitelist MIME types: `["application/pdf", "application/msword", "application/vnd.openxmlformats-officedocument.wordprocessingml.document"]`.
   122→  2. Enforce 5 MB max size: `if (resumeFile.size > 5 * 1024 * 1024) return fail("Resume must be ≤ 5 MB", 413);`
   123→  3. Validate file signature (magic bytes), not just the declared MIME.
   124→  4. Store the file in object storage (S3 / OSS / local disk under `/uploads/`) — never in the database.
   125→  5. Generate an unguessable filename (e.g., `crypto.randomUUID() + ext`) and serve via a streaming endpoint with `Content-Disposition: attachment`.
   126→  6. Add a cleanup cron to delete unclaimed resumes after 90 days.

   147→### C6. No Next.js App Router error boundaries (`error.tsx`, `global-error.tsx`, `loading.tsx`)

   156→### C7. No `middleware.ts` — admin API routes have no defense-in-depth guard
   157→- **Files**: missing — `src/middleware.ts`
   158→- **Problem**: All `/api/admin/*` routes call `requireAdmin()` inside the handler (good), but there is no middleware-level guard. This means:
   159→  - Every admin route pays the cost of `getServerSession()` on every request.
   160→  - There is no centralized rate limiting for admin endpoints (only the public `contact`/`apply`/`newsletter` routes are rate-limited).
   161→  - There is no CSRF protection on session-cookie-authenticated mutations.
   162→- **Impact**: Brute-force and CSRF attacks against admin endpoints are easier than necessary.
   163→- **Fix**:
   164→  1. Add `src/middleware.ts` that intercepts `/api/admin/*` and short-circuits with 401 if no session cookie is present.
   165→  2. Add rate-limiting for admin login (`/api/auth/[...nextauth]`) — currently unthrottled.
   166→  3. Add CSRF token check for all mutation routes (or use `SameSite=Strict` cookies).

   168→### C8. In-memory rate limiter does not survive restarts and is per-process
   169→- **File**: `src/lib/rate-limit.ts:11`
   170→- **Problem**:
   171→  ```ts
   172→  const buckets = new Map<string, Bucket>();
   173→  ```
   174→  The token-bucket map lives in module memory. Under serverless/standalone deployment with multiple workers (or restarts), each process has its own map — an attacker distributing requests across workers gets N× the rate limit. A restart wipes all buckets.
   175→- **Impact**: Rate limits are ineffective in production multi-process deployments. The contact/apply/newsletter endpoints are effectively unthrottled.
   176→- **Fix**: Move rate-limit state to a shared store:
   177→  - **Simpler**: Upstash Redis with `@upstash/ratelimit` (serverless-friendly).
   178→  - **Self-hosted**: `ioredis` against a Redis container.
   179→  - **Single-process workaround (NOT recommended for prod)**: document that the deployment MUST be single-process and set `NEXT_PRIVATE_WORKER_COUNT=1`.

   210→### H4. No pagination on admin list endpoints — returns ALL records
   211→- **Files**: `src/lib/crud-factory.ts:22-30` (`makeListHandler`), `src/app/api/admin/leads/route.ts:18-22`, `src/app/api/admin/applications/route.ts:17-22`, `src/app/api/admin/activity/route.ts:13-17`

   216→### H5. Public search endpoint has no `take` limit and no input length cap
   217→- **File**: `src/app/api/search/route.ts`

   222→### H6. `crud-factory.ts` uses `args?: any` for Prisma delegates
   223→- **File**: `src/lib/crud-factory.ts:13,33,96-98`
   224→- **Problem**: The Prisma-delegate types are hand-rolled with `any` to dodge Prisma's complex generic delegate types. This loses type-safety on `findMany`, `create`, `update`, `delete` arguments and return types.
   225→- **Fix**: Use Prisma's built-in delegate types — `Prisma.ProjectDelegate`, `Prisma.ServiceDelegate`, etc. — or generic `Prisma.ModelDelegate<T>` via `db[Model]`. This is the single biggest type-safety win in the codebase.

   227→### H7. `ReactMarkdown` renders user-controlled content without URL sanitization
   228→- **File**: `src/components/views/news-view.tsx:368`
   229→- **Problem**: Blog post `content` (markdown authored by admins) is rendered with `<ReactMarkdown components={markdownHeadingComponents}>`. By default `react-markdown` does NOT render raw HTML (no `rehype-raw`), so XSS via inline HTML is blocked. **However**, there is no `remark-gfm` URL sanitization and no `allowedSchemes` filter — a markdown link like `[click](javascript:alert(1))` would be rendered as `<a href="javascript:alert(1)">`. (Recent `react-markdown` versions DO filter `javascript:` URLs, but it's worth being explicit.)
   230→- **Fix**: Add explicit URL sanitization:
   231→  ```ts
   232→  <ReactMarkdown
   233→    components={markdownHeadingComponents}
   234→    urlTransform={(url) => /^https?:/i.test(url) || url.startsWith("#") ? url : ""}
   235→  >
   236→  ```

   238→### H8. `useSiteData` refetches on every mount; no SSR / no initial data
   239→- **File**: `src/lib/hooks/use-queries.ts:32-38`
   240→- **Problem**: `staleTime: 5 * 60 * 1000` is OK, but every component mounting calls `useSiteData()` — there's no `initialData` and no SSR. The home view shows skeletons for ~500 ms on every cold load. Combined with the fact that all views are client components, this defeats Next.js's RSC data-fetching.
   241→- **Fix**: Convert the home view (at minimum) to a Server Component that fetches site data with `fetch()` + `next: { revalidate: 300 }` and passes it as props. Hydrate the React Query cache via `<HydrationBoundary state={dehydrate(queryClient)}>`. See Architecture Recommendations §B.

   243→### H9. Public API routes have no `Cache-Control` / `revalidate` headers
   244→- **Files**: `src/app/api/public/site/route.ts`, `src/app/api/public/projects/route.ts`, `src/app/api/public/blog/route.ts`, etc.
   245→- **Problem**: Every public GET endpoint hits the database on every request. Next.js does not cache route handlers by default.

   247→  - Add `export const revalidate = 300;` to each route (Full Route Cache), OR
   248→  - Wrap responses with `NR.json(data, { headers: { "Cache-Control": "s-maxage=300, stale-while-revalidate=86400" } })`.

   250→### H10. Resume download in admin uses raw `<a href>` on a data URL
   251→- **File**: `src/components/admin/panels/applications.tsx:285`
   252→- **Problem**: `<a href={app.resumeUrl as string} download>` where `resumeUrl` is a base64 data URL. This works for small PDFs but:
   253→  - Bloated URLs (multi-MB base64 strings) freeze the browser tab.
   254→  - No content-type sniffing.
   255→  - Vulnerable to stored XSS if a `data:text/html` payload sneaks through (see C4).
   256→- **Fix**: After implementing C4 (object storage), link to a streaming download endpoint `/api/admin/applications/[id]/resume` that sets `Content-Disposition: attachment` and streams the file.

   258→### H11. Single-route Zustand architecture is a major SEO/a11y/UX regression
   259→- **Files**: `src/lib/store.ts`, `src/components/site/view-router.tsx`, `src/app/page.tsx`, `src/app/sitemap.ts`
   260→- **Problem**: The entire site is one `/` route. "Navigation" calls `setView()` which mutates Zustand state — the URL **never changes**. Consequences:
   261→  - **No deep linking**: a user on `/about` who refreshes lands back on home.
   262→  - **No browser back/forward**: the browser history has only one entry. (Worklog claims smooth view transitions, but the user loses their place on back.)
   263→  - **SEO**: `sitemap.ts` lists 12 views but they all map to the same URL (`https://mhaksa.com`). Google sees one page. The "priorities" in the sitemap are useless.
   264→  - **Analytics**: cannot tell which "page" a visitor is on without custom events.
   265→  - **Sharing**: cannot share a link to a specific project, blog post, or job.
   266→  - **Accessibility**: screen readers cannot use `aria-current="page"` properly; the document title never changes.
   267→  - **Code splitting**: ViewRouter lazy-loads views, but without URL routing there's no `next/link` prefetching.
   268→- **Impact**: Severe — defeats the entire purpose of using Next.js App Router.
   269→- **Recommendation**: Implement proper multi-route App Router architecture. See "Architecture Recommendations §A" below.

   409→## Architecture Recommendations
   410→
   411→### §A. Migrate from Zustand view-state to proper Next.js App Router routing
   412→
   413→**Current state**: Single `/` route, Zustand holds `view: "home" | "about" | ...`, navigation calls `setView()`, URL never changes.
   414→
   415→**Recommendation**: This is the single most impactful architectural change. The "user can only see the `/` route" sandbox constraint noted in the worklog appears to be a **sandbox restriction, not a product requirement**. For a production corporate website, proper URL routing is non-negotiable for the reasons listed in H11.
   416→
   417→**Proposed route structure**:
   418→```
   419→src/app/
   420→├── layout.tsx                    # Root layout (server component)
   421→├── page.tsx                      # Home (/)
   422→├── about/page.tsx                # /about
   423→├── services/
   424→│   ├── page.tsx                  # /services
   425→│   └── [slug]/page.tsx           # /services/[slug]   (generateStaticParams)
   426→├── projects/
   427→│   ├── page.tsx                  # /projects
   428→│   └── [slug]/page.tsx           # /projects/[slug]   (generateStaticParams)
   429→├── gallery/page.tsx              # /gallery
   430→├── clients/page.tsx              # /clients
   431→├── careers/
   432→│   ├── page.tsx                  # /careers
   433→│   └── [slug]/page.tsx           # /careers/[slug]   (generateStaticParams)
   434→├── news/
   435→│   ├── page.tsx                  # /news
   436→│   └── [slug]/page.tsx           # /news/[slug]   (generateStaticParams)
   437→├── faq/page.tsx                  # /faq
   438→├── contact/page.tsx              # /contact
   439→├── legal/
   440→│   ├── privacy/page.tsx          # /legal/privacy
   441→│   └── terms/page.tsx            # /legal/terms
   442→├── admin/
   443→│   ├── layout.tsx                # Auth-gated admin layout (server)
   444→│   └── page.tsx                  # /admin dashboard
   445→└── api/...
   446→```
   447→
   448→**Migration approach** (low-risk, incremental):
   449→1. **Phase 1** — Keep Zustand as a *secondary* navigation hint; add a `useEffect` in `RtlRoot` that syncs `window.location.pathname` → Zustand `view` on back/forward (popstate). Add `history.pushState` on `setView`. This restores back-button without rewriting anything.
   450→2. **Phase 2** — Create new App Router pages that render the same view components (e.g., `app/about/page.tsx` → `<AboutView />`). Both the Zustand-driven `/` and the new `/about` work.
   451→3. **Phase 3** — Convert navigation `setView("about")` calls to `<Link href="/about">`. Update `Header`/`Footer` nav.
   452→4. **Phase 4** — Move data fetching into Server Components; use `generateStaticParams` for `[slug]` routes; remove TanStack Query for public read-only data (keep it for admin).
   453→5. **Phase 5** — Add `[locale]` segment for proper `/en/...` and `/ar/...` bilingual routing using `next-intl` (already installed but unused).
   454→
   455→**Effort estimate**: 3–5 days for a senior engineer.
   456→
   457→### §B. Adopt Server Components for public pages
   458→
   459→Currently **every** view component is `"use client"`. This means:
   460→- No SSR data fetching — every page shows a skeleton for ~500 ms.
   461→- No SEO benefits from server-rendered content.
   462→- Larger JS bundle (TanStack Query, all view code ships to the client).
   463→
   464→**Recommendation**: Convert public views to Server Components that fetch via Prisma directly (server-side) and pass typed props to client sub-components. Keep client components only where interactivity is required (forms, carousels, animations).
   465→
   466→### §C. Replace `crud-factory.ts`'s `any` with proper Prisma types
   467→
   468→Use Prisma's generated delegate types (`Prisma.ServiceDelegate`, etc.) or a generic helper that preserves type inference. This eliminates 5+ `as any`/`as Record<string, unknown>` casts per route file.
   469→
   470→### §D. Adopt a transactional outbox for activity logging
   471→
   472→Currently `logActivity` is fire-and-forget with `catch(() => {})`. If the DB write fails (e.g., DB down, constraint violation), the audit trail is silently lost. For an enterprise audit trail, use a transactional outbox pattern or at minimum log failures to a fallback sink (stderr + Sentry).
   473→
   474→### §E. Add observability: Sentry + structured logging
   475→
   476→There is currently no error monitoring, no structured logging, no request tracing. For a production enterprise app, add:
   477→- `@sentry/nextjs` for error tracking.
   478→- `pino` or `winston` for structured server logs.
   479→- A `requestId` propagated via `X-Request-ID` header for tracing.
   480→
   481→### §F. Replace SQLite with PostgreSQL for production
   482→
   483→SQLite is fine for development but problematic for production multi-process deployments (file locking, no concurrent writes). The Prisma schema is DB-agnostic — switching to Postgres is a `provider = "postgresql"` change + a `DATABASE_URL` swap. This also enables `Json` column types (fixes M14), proper full-text search (better than SQLite `contains`), and native UUID.
